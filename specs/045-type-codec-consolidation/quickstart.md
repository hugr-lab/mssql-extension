# Quickstart: Type Codec Consolidation (spec 045)

A practical guide for two audiences:

1. **Reviewers** — to navigate the spec-045 PR and locate the change at
   each of the 5 dispatch sites and inside each of the 9 family modules.
2. **Future implementers** — to add a new type family (e.g., for a
   hypothetical SQL Server `JSON` native type) or fix a bug in an
   existing family without re-learning the dispatch topology.

This file is **post-implementation reading**. It assumes the per-family
modules have landed under `src/codec/`. During implementation, the
phased migration order is defined in tasks.md (generated by
`/speckit-tasks`); this file describes the **end-state** layout.

## "I want to add support for a new SQL Server type"

Suppose SQL Server ships a hypothetical native `JSON` type (TDS id
`0xC0`, mapping to DuckDB `JSON` which is currently aliased to VARCHAR).

### Pre-spec-045 (the old way)

You would edit **5 files**:

1. `src/tds/encoding/type_converter.cpp` — add `case TDS_TYPE_JSON:` arm,
   write a `ConvertJson` helper.
2. `src/tds/encoding/bcp_row_encoder.cpp` — add `case LogicalTypeId::JSON:`
   arm, write an `EncodeJson` helper.
3. `src/table_scan/filter_encoder.cpp` — add a `case LogicalTypeId::JSON:`
   arm to `ValueToSQLLiteral`.
4. `src/dml/insert/mssql_value_serializer.cpp` — add a
   `case LogicalTypeId::JSON:` arm to `Serialize`.
5. `src/catalog/mssql_ddl_translator.cpp` — add `case LogicalTypeId::JSON:`
   arms to BOTH `MapTypeToSQLServer` and `MapLogicalTypeToCTAS`.

Five files. Five switches. Five places to forget to update.

### Post-spec-045 (the new way)

You edit **one new file pair** + add 2 lines to the dispatch helpers:

1. Create `src/codec/json_codec.cpp` + `src/include/codec/json_codec.hpp`.
   Implement the four standard ops (`DecodeFromTds`, `EncodeToBcp`,
   `FormatSqlLiteral`, `FormatDdlTypeName`).
2. Add `Json,` to `TypeFamily` enum in `src/include/codec/type_family.hpp`.
3. Add `case TDS_TYPE_JSON: return TypeFamily::Json;` to
   `FamilyFromTdsType` in `src/codec/type_family.cpp`.
4. Add `case LogicalTypeId::JSON: return TypeFamily::Json;` to
   `FamilyFromLogicalType` in `src/codec/type_family.cpp`.
5. Add a new `case TypeFamily::Json:` arm to each of the 5 dispatch
   sites' switches (one line each, calling `codec::json::<op>(...)`).
6. Add `test/cpp/codec/test_json_codec.cpp` with golden fixtures.

The dispatch sites grow by 5 one-liner cases. All the actual JSON
encoding/decoding/literal/DDL logic lives in **one file**.

## "I want to fix a bug in how DECIMAL is BCP-encoded"

Pre-spec-045: open `src/tds/encoding/bcp_row_encoder.cpp`, find
`EncodeDecimal` (line 365), make the change. Then check that
`mssql_value_serializer.cpp:Serialize`'s DECIMAL arm doesn't have a
parallel bug, and that `filter_encoder.cpp:ValueToSQLLiteral`'s DECIMAL
arm doesn't either. Three places to inspect.

Post-spec-045: open `src/codec/decimal_codec.cpp`. All four DECIMAL
operations are there. Fix in one place. Reviewers know exactly where
to look.

## "I want to add a new T-SQL literal escape rule for VARCHAR"

Pre-spec-045: `src/table_scan/filter_encoder.cpp:ValueToSQLLiteral`'s
VARCHAR arm uses `EscapeStringLiteral`. `src/dml/insert/mssql_value_serializer.cpp:Serialize`'s
VARCHAR arm calls `SerializeString(StringValue::Get(value))`. The two
escape paths are subtly different. Updating both correctly requires
finding both.

Post-spec-045: open `src/codec/string_codec.cpp`. `FormatSqlLiteral`'s
VARCHAR arm handles the escape rule once. Both consumers
(`FilterEncoder::ValueToSQLLiteral` and `MSSQLValueSerializer::Serialize`)
already go through this function.

If a context needs a different escape rule (rare — none exist today),
branch on `LiteralContext` inside the VARCHAR arm and document why.

## "I want to update the SQL Server DDL type-name for HUGEINT"

Pre-spec-045: open `src/catalog/mssql_ddl_translator.cpp`. Find
`MapTypeToSQLServer` (line 78). HUGEINT returns `"DECIMAL(38,0)"`. Find
`MapLogicalTypeToCTAS` (line 351). HUGEINT throws. These are two
different files^Wfunctions with subtly divergent behavior.

Post-spec-045: open `src/codec/integer_codec.cpp`. `FormatDdlTypeName`
has a `DdlContext` branch:
- `DdlContext::CreateTable` → return `"DECIMAL(38,0)"`
- `DdlContext::CtasCreateTable` → `throw NotImplementedException(...)`

The divergence is explicit in one place. To unify (e.g., have CTAS also
return `"DECIMAL(38,0)"`), delete the `throw` branch. One edit.

## "I want to verify the spec-045 dispatch refactor didn't change behavior"

Two layers of guards:

1. **Per-family golden fixtures** (`specs/045-type-codec-consolidation/golden/<family>/`):
   each family migration captures pre-migration outputs and asserts
   the post-migration code produces byte-identical results.
2. **SQL test suite** (`make test` + `make integration-test`): the 103+
   test cases / 3229+ assertions all stay green per FR-032.

The only intentional behavior changes are the three documented fixes:
- **Issue #91** (`test/sql/copy/copy_nvarchar_length_validation.test`)
- **HUGEINT filter literal** (`test/sql/catalog/filter_pushdown_hugeint.test`)
- **DECIMAL filter literal** (`test/sql/catalog/filter_pushdown_decimal.test`)

Each of these regression tests **fails on `main`-at-spec-045-kickoff**
(proving the bug was real) and **passes on the spec-045 PR HEAD** (per
SC-002a). The pre/post byte traces are captured under
`specs/045-type-codec-consolidation/issue_91_repro.md` and
`...filter_pushdown_*_repro.md`.

## "I want to understand why Money is in a special position"

Money is **scan-decode-only** because:
- SQL Server `MONEY` / `SMALLMONEY` columns exist and arrive on the
  TDS wire as Money TDS types — so `codec::money::DecodeFromTds` IS
  implemented (decodes to DuckDB `DECIMAL(19,4)` / `DECIMAL(10,4)`).
- DuckDB has no MONEY type. So when DuckDB sends data back to SQL
  Server (BCP encode, INSERT literal, DDL type name), the value is a
  `DECIMAL` — it routes through `codec::decimal::*`, NEVER through
  `codec::money::*`.
- Therefore `codec::money::EncodeToBcp`, `FormatSqlLiteral`,
  `FormatDdlTypeName` are **not implemented**. The file pair contains
  only `DecodeFromTds`.

This asymmetry is documented in `src/codec/money_codec.hpp` and in
data-model.md.

## Quick-reference: who calls whom

| Caller (dispatch site)                                  | Calls                                                              |
|---------------------------------------------------------|--------------------------------------------------------------------|
| `type_converter.cpp:ConvertValue`                       | `codec::<family>::DecodeFromTds(...)`                              |
| `bcp_row_encoder.cpp:EncodeRow`                         | `codec::<family>::EncodeToBcp(...)`                                |
| `filter_encoder.cpp:ValueToSQLLiteral`                  | `codec::FormatSqlLiteral(v, type, LiteralContext::Filter)`         |
| `mssql_value_serializer.cpp:Serialize`                  | `codec::FormatSqlLiteral(v, type, LiteralContext::InsertValues)`   |
| `mssql_ddl_translator.cpp:MapTypeToSQLServer`           | `codec::<family>::FormatDdlTypeName(type, cfg, DdlContext::CreateTable)`        |
| `mssql_ddl_translator.cpp:MapLogicalTypeToCTAS`         | `codec::<family>::FormatDdlTypeName(type, cfg, DdlContext::CtasCreateTable)`    |
| `codec::FormatSqlLiteral` (shared dispatcher)           | `codec::<family>::FormatSqlLiteral(...)`                           |

That's the entire surface area.
